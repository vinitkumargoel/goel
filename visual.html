Feasibility & UI study

Download straight to an SFTP server

You start a download, and instead of picking a folder on your Mac you pick a saved SFTP server and a remote directory. The file lands there. Your Mac stays clean.

✓

Feasible — and most of the machinery already exists

You already ship a full SFTP client, saved servers with Keychain credentials and host-key pinning, a remote file browser, and a transfer engine with progress/cancel/retry. The honest answer is that “never touches my disk” is only achievable for some download types. The design below ships the useful 100% via a staged relay, and adds true zero-disk streaming for the subset where it actually works — without ever lying to the user about which one they’re getting.

1 · The idea, restated

Today the Add-download sheet’s “Save to” dropdown offers ~/Downloads, ~/Movies, “Automatic (by type)” and “Choose folder…”. Every one of those is a path on this Mac. You want a fourth kind of answer: a server.

The motivating use cases, in rough order of how well this design serves them:

  • Home server / NAS as the real destination. Grab a 40 GB Linux ISO or a video and have it end up on the box that actually stores things, not on a laptop SSD you then have to clean up.
  • Small local disk. The Mac physically cannot hold what you’re fetching.
  • Skip the round trip. Downloading to the Mac and then uploading is two transfers and two waits — you want one action.

Those three want different things, and the difference matters. Case 3 is satisfied by any design that automates the two hops. Case 1 mostly is too. Only case 2 genuinely requires that the bytes never land locally — and that is the expensive one. The design below is built so cases 1 and 3 work everywhere and case 2 works where it physically can.

2 · Three ways to build it

A · Stream-through

Bytes go from the HTTP socket into a small bounded buffer and straight out over libssh2_sftp_write. Nothing is written to local disk, ever.

True zero-disk No resume No segmentation

B · Stage & forward

Download normally into a staging folder at full segmented speed, then upload to the server as a post-completion step, then delete the local copy.

Works for everything Full resume Needs transient disk

C · Remote-side fetch

Open an SSH exec channel and run curl on the server itself. The Mac only supervises; the bytes never cross your network at all.

Fastest by far Needs shell + curl Out of scope

How each fares per download type

Download typeA · Stream-throughB · Stage & forwardWhy
HTTP, single connection Works Works A sequential byte stream is exactly what the SFTP write loop wants.
HTTP, segmented
(the default, and the fast path)		 Blocked Works Segments seek() to their own offset in a pre-sized file. The SFTP shim opens with FXF_TRUNC and only appends — there is no seek, and no second write handle.
FTP Works Works Sequential stream, same as single-connection HTTP.
SFTP → SFTP Works Works Two sessions, one pumping the other. Genuinely useful: server-to-server copy through your Mac.
HLS video Blocked Works Segments download concurrently into a work dir, then concatenate and remux to .mp4 through AVFoundation. The muxer needs a real local file.
BitTorrent Impossible Works libtorrent owns its own storage and writes pieces in random order as they arrive. There is no byte stream to intercept.

What you give up with each

CapabilityA · Stream-throughB · Stage & forward
Pause / resume Lost — the shim truncates on open, so any interruption restarts from byte 0. On a 40 GB file over a flaky link that is close to unusable. Kept — normal download resume; a failed upload retries from a local file that still exists.
Speed Capped at the slower of the two links, single-connection, no acceleration. Typically 2–5× slower than segmented on a fast source. Full download speed, then upload at LAN speed. Two phases, but each at maximum.
Checksum verification Rolling hash only — computable in-flight, but a mismatch is discovered after a bad file already sits on the server. Kept — verify locally, refuse to upload on mismatch. Nothing bad ever reaches the server.
Antivirus / auto-extract / post-download script Lost — all three take a local file path. Kept — they run on the staged file before it is sent.
Local disk needed A few MB of buffer. Full file size, transiently.
Implementation cost Producer/consumer bridge with backpressure across a C thread boundary. Fiddly, and easy to deadlock. A new stage bolted onto the existing post-completion pipeline. Largely assembling parts you already have.

The uncomfortable finding. Mode A — the one that literally matches “it will not come to my machine” — is the weaker feature. It only covers the download types you use least, it forfeits resume, it is slower, and it can put a corrupt file on your server. It is worth building, but as a deliberate opt-in for the small-disk case, not as the headline.

Why C is out of scope (for now)

Running curl on the server is the fastest possible answer and the one a home-server owner would eventually want. But the C shim exposes the SFTP subsystem only — there is no exec channel — and adding one means shipping remote command execution, inheriting whatever curl is or isn’t installed, losing per-segment progress, and taking on a materially larger security surface. Worth revisiting once A and B are in and you know whether you actually reach for it.

3 · What the code actually allows

Everything below was read out of the repo, not assumed. These are the facts the design has to bend around.

The blockers

Blocker  The SFTP upload path truncates and cannot seek

gsb_upload opens the remote file with LIBSSH2_FXF_WRITE | LIBSSH2_FXF_CREAT | LIBSSH2_FXF_TRUNC and then runs a single sequential pull loop. There is no offset parameter, no libssh2_sftp_seek64, and one write handle.

Consequences: segmented download can’t stream out; upload resume is impossible; a retry always restarts at zero.
Source: Sources/SSHBridge/ssh_bridge.c:317–371, Sources/SSHBridge/include/ssh_bridge.h:93

Blocker  Segmented HTTP writes at offsets into a pre-sized file

The transfer pre-sizes the destination, then each segment opens a handle and seek(toOffset:)s to its own start before writing — precisely so segments don’t race to grow the file. That model has no remote equivalent through the current shim.
Source: Sources/GoelCore/Engine/SegmentedTransfer.swift:185, 264, 766–773

Blocker  Torrent and HLS own their local storage

libtorrent is handed task.saveDirectory and writes pieces itself in arrival order. HLS pulls segments concurrently into a work dir, concatenates them, and remuxes MPEG-TS to .mp4 via AVFoundation. Neither exposes an interceptable sequential stream.
Source: TorrentEngine.swift:243–246, HLSEngine.swift:14–20, 189–222

Blocker  A background sweep would silently delete these tasks

Every 5 seconds, the file-reconcile pass prunes any .completed task whose saveDirectory exists but whose savePath does not — the “user deleted the file in Finder” cleanup. A task whose payload now lives only on a server matches that condition exactly and would vanish from the list within five seconds of finishing.
Source: Sources/GoelCore/Scheduler/DownloadManager+FileReconcile.swift:19–66

The same local-path assumption sits in FileProgressPublisher (skips tasks with no local file), rename (moveItem within saveDirectory), open/reveal/play, and the remote-control server’s file streaming. Each needs a remote-destination branch — this is the quiet, unglamorous half of the work.

Friction  saveDirectory is a bare local String, used in 16 files

DownloadTask.saveDirectory: String feeds savePath, primaryFilePath, the PathSafety containment guard, persistence, the remote-control router and every engine — 61 references across 16 files. Turning it into an enum would touch all of them.
Source: Sources/GoelCore/Model/DownloadTask.swift:12, 261–293

The way through: don’t. Leave saveDirectory meaning “where the bytes live locally” and add a separate optional remoteDestination. During staging every existing path-based behaviour keeps working untouched. See §5.

The gifts

Have  Saved servers, fully modelled

SFTPConnection with host/port/user/initial path, passwords in the Keychain keyed by user@host:port, trust-on-first-use host-key pinning via HostKeyStore, and a sidebar SERVERS section that already exists.

Have  Live reachability, without spending credentials

ServerReachability probes with a plain TCP connect and DNS resolve — deliberately unauthenticated — and carries latency, IP and detected OS. A destination picker can show online/offline dots for free.

Have — but in the wrong layer  A working upload engine

SFTPTransfer has progress, cancel, retry, folder recursion, a progress coalescer and a conflict sheet. But it lives on AppViewModel in GoelApp, is ad-hoc, unqueued and unpersisted — while the upload stage needs to run in GoelCore beside the scheduler. The logic transfers; the code has to move down a layer first. See §5.

Have  A post-completion pipeline with a seam

Completion already chains checksum verify → antivirus scan → extract/script, each gated on the previous. “Upload, then delete local” is one more link on the end of a chain that already exists.

Have  A remote file browser

SFTPBrowserView lists, navigates, creates directories and renames. A “pick a folder” mode is a presentation variant, not a new feature.

Have  A pull-based upload loop

gsb_upload takes a read_cb rather than a file descriptor. Mode A can feed it from a memory ring buffer with no C changes at all — the streaming door is already open.

Two small C additions worth making

  • gsb_statvfs — libssh2_sftp_statvfs() exists and is not yet wrapped. Wrapping it gives a real remote free-space preflight instead of a guess.
  • An append/offset flag on gsb_upload — swapping FXF_TRUNC for FXF_APPEND plus a seek64 would restore upload resume. Not needed for v1; it is what unlocks resumable remote uploads later.

4 · The recommended design

Ship Mode B as the default and the headline. Offer Mode A as an explicit per-task opt-in, only when it is actually eligible. One destination picker, two delivery modes, and copy that never overstates which one you’re getting.

Mode B — “Relay via this Mac” (default)

source ──▶ staging folder on this Mac ──▶ verify ─▶ scan ──▶ SFTP upload ──▶ remote server │ └─▶ local copy deleted (or kept, if you ask)
  • Downloads at full segmented speed with resume, into an app-owned staging directory.
  • Checksum and antivirus run before the upload, so a bad file never reaches the server.
  • Uploads to name.part, then rename()s into place — a killed upload leaves an obviously-partial file, never a plausible-looking truncated one.
  • Local copy is removed on a verified upload. A “keep a local copy too” toggle turns it into a mirror.
  • Works for every download type, including torrents.

Mode A — “Stream directly (no local copy)”

source ──▶ bounded ring buffer (~8 MB) ──▶ remote server no local file · no resume · no antivirus · single connection
  • Offered only for single-file HTTP, FTP and SFTP→SFTP sources.
  • Forces single-connection transfer; the segmented path is disabled for the task.
  • Pause is not offered — only cancel, which discards everything. The UI must say so up front.
  • Checksum, if given, is computed rolling and reported after the fact; a mismatch marks the task failed and offers to delete the remote file.

The rule that keeps it honest

The mode selector is never silently downgraded. If you pick “Stream directly” and then change the source to a magnet link, the chip becomes disabled with a visible reason (screen 08) rather than quietly falling back to relay. Anything else and the one property you cared about — bytes not touching your disk — becomes something you can’t trust.

5 · Data model & blast radius

The additive change

struct RemoteDestination: Codable, Sendable, Hashable { var connectionID: UUID // → SFTPConnection in the store var remoteDirectory: String // server-absolute, or "." for home var mode: DeliveryMode // .relay | .stream var keepLocalCopy: Bool var onConflict: ConflictPolicy // .ask | .overwrite | .rename | .skip } // DownloadTask gains exactly one optional field: var remoteDestination: RemoteDestination? // nil = today's behaviour, unchanged

saveDirectory keeps its current meaning and its 61 call sites keep working. When a remote destination is set it points at the staging directory instead of ~/Downloads. Every engine, the PathSafety guard, primaryFilePath, the remote-control router and the detail panel all carry on operating on a real local path for the whole download phase. The remote destination only becomes relevant once the bytes are complete.

Because the field is optional, existing persisted blobs decode unchanged — the same pattern already used for tags, mirrors and seedRatioLimit. No migration.

New status states

DownloadStatus gains .uploading alongside the existing .verifying, and DownloadError gains a few remote-specific cases so failures read as something you can act on rather than “Unknown error”:

case uploading // bytes complete locally, sending to the server case remoteAuthFailed(String) // password rejected / agent had no identity case remoteHostKeyMismatch // pinned fingerprint changed — hard stop case remoteDiskFull(needed:available:) case remotePermissionDenied(String) case remoteUnreachable(String)

Host key mismatch — hard stop

security · deliberately obstructive

The one place in this feature where the UI should be uncooperative. HostKeyStore pins on first connect; a changed fingerprint means either a legitimate rebuild or an active interception, and the app cannot tell which. There is no “continue anyway” button — resolving it requires going to the server’s settings and explicitly re-trusting, which is enough friction to force a moment of thought.

Deliberate omission. No “Trust this key and continue” in the failure dialog. Re-trusting is possible, but only from the server’s own settings, away from the pressure of a stalled 4.7 GB upload. A one-click trust button here would be clicked reflexively every single time.

Paused during upload

edge state · honest labelling

A subtle one worth getting right. Pausing during the upload phase cannot be a true pause — the shim truncates on reopen, so resuming means re-sending from zero. The control is therefore labelled “Hold upload” and the note says what resuming will actually do. Calling it “Pause” and then silently restarting a 4 GB transfer is the sort of small lie that erodes trust in the whole app.

7 · Risks worth naming

RiskSeverityMitigation
Staging fills the boot diskHigh Preflight before queueing (screen 11); a running total of staged bytes across all pending uploads, not just this one; a configurable staging cap that pauses new remote-destination downloads when hit.
Orphaned staged filesMedium A crash mid-upload leaves a full-size file in staging. Reconcile at launch against persisted tasks and surface anything unclaimed as reclaimable space rather than deleting silently.
Long uploads outlive the SSH sessionMedium A multi-hour upload can hit the server’s ClientAliveInterval. Needs libssh2 keepalive enabled on the session — currently not configured anywhere in the shim.
No connection poolingMedium Every operation opens a fresh TCP + SSH handshake + auth and tears it down. A destination card that checks reachability, lists a folder and reads free space is three full handshakes. Batch them into one session, and never trigger them from hover.
Staging claims full size immediatelyMedium Segmented HTTP preallocates the destination to its full length up front, so a 154 GB download occupies 154 GB from second one — not gradually. The preflight in screen 11 is therefore a hard gate, not a courtesy.
Partial .part files accumulateMedium Upload to .part + rename(); on any failure offer explicit removal (screen 16); sweep known-stale .part files on reconnect.
Credential spend on hoverMedium The dropdown must use the existing unauthenticated TCP probe only. Remote free space and folder listing need a real session, so they happen on explicit selection or Browse — never on hover or menu open.
Mode A deadlockHigh A ring buffer bridging an async Swift producer to a blocking C consumer on another thread is the classic place to hang forever. Needs a hard timeout on both sides and a bounded buffer with explicit backpressure, plus a test that kills each side mid-transfer.
Torrent seeding vs. cleanupMedium Deleting the staged copy stops seeding. Default to keeping it until seeding ends; make it a visible setting (see §9).
Remote-control / web-portal driftMedium RemoteRouter and the Linux daemon read saveDirectory. They must either understand remote destinations or reject them with a clear error — silently writing to a local path would be the worst outcome.
Scope creep into “remote download manager”Medium Once downloads can target a server, “start it from my phone” and “run it on the server” follow naturally. Worth deciding deliberately rather than drifting there.

8 · Phasing

Each phase is independently shippable and independently useful.

Phase 1  Relay to a saved server  — the whole feature for most people

RemoteDestination model · staging management · upload stage on the completion pipeline · teaching FileReconcile and the other local-path assumptions about remote tasks · the Save-to dropdown and destination card · the remote folder picker · two-phase progress in list and detail · conflict handling reusing the existing sheet · the failure states.
Screens: 01, 02, 04, 05, 06, 07, 09, 10, 13, 14, 16, 18, 20, 21.

Phase 2  Preflight & defaults

The gsb_statvfs wrapper for real remote free space · local staging preflight · the Destinations settings pane · per-server default folders · unreachable-server retry policy.
Screens: 11 (relay half), 12, 19.

Phase 3  True direct streaming

The bounded ring-buffer sink · forcing single-connection transfer · eligibility rules and the mode chip · rolling checksum · the “can’t resume” consequences throughout.
Screens: 03, 08, 11 (stream half), 17.
The hardest phase and the least broadly useful. Worth doing only if Phase 1 lands and the small-disk case turns out to be real for you.

Later  Resumable remote uploads · remote-side fetch

Adding FXF_APPEND + seek64 to the shim removes the biggest wart in both modes and would let screen 18 become a real pause. Architecture C (running curl on the server) only makes sense once you know you want it.

9 · Open questions for you

  1. Which case is actually yours? Home-server-as-destination (Phase 1 is enough), or a Mac that genuinely can’t hold the file (you need Phase 3)? This single answer changes the whole plan’s shape.
  2. Torrents and seeding. Keep seeding from the staged copy until the ratio target is met, or upload and stop seeding immediately? The first is friendlier, the second is what “don’t keep it on my machine” literally asks for.
  3. Multi-file torrents. Upload the whole folder tree preserving structure, or flatten? The existing folder-upload code recurses, so preserving structure is nearly free.
  4. Should a destination be settable after the fact? Right-click a completed local download → “Send to prod-box” is a small addition on the same plumbing and might be more useful day to day than the add-time picker.
  5. Linux daemon parity. Does GoelDaemon need this too, or is it macOS-only for now? The web portal would need the same picker.
  6. Watch folders and RSS. Should automation rules be able to target a server destination, or is this add-sheet-only in v1?
