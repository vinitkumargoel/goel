One — the core loop

The queue is the app. Everything here is real engine state — segmented multi-connection transfer, live speed and ETA, mirror failover, and sequential download that makes a file playable long before it completes.

Two — glanceable surfaces

The promise is "don't babysit it." These are what make that believable.

Three different technologies with three different update models — conflating them is how this ships broken. The Live Activity is live; the widgets are ambient and refresh on a budget. They are scoped so they can never visibly contradict each other.

Three — the rest of the app

Files you can actually reach, and controls that respect your data plan.

iOS has no filesystem to write to, so the container becomes a first-class library that also appears in Files. Settings carry the engine's real capabilities — traffic profiles, per-task caps, scheduling — without exposing desktop concepts that mean nothing on a phone.

On fidelity. Frames are drawn at 1 pt = 1 px against iPhone 15/16 Pro metrics (393 × 852 pt, 55 pt corner, 125 × 36 pt Dynamic Island, 59/34 pt safe areas) using iOS system colours and the `-apple-system` stack — so on an Apple display this is genuinely SF Pro, not an approximation. What HTML cannot reproduce: true SF Symbols (icons here are hand-drawn SVG equivalents), UIKit material blur, spring physics, and Dynamic Type. Treat this as a high-fidelity visual target, not a rendering of shipped code.

Honest about the hard part. The backgrounded Dynamic Island state is drawn deliberately degraded. While the app is suspended iOS runs the transfer out-of-process, so a Live Activity cannot smoothly animate without a push server — a dependency this product rejects. Showing a byte count and “updated 2 min ago” is the truthful design; a confident percentage would be a lie.
