Feature proposal · plain-language summary

Sending downloads straight to a server

Today, everything you download lands on your Mac. This proposal lets you pick one of your saved servers as the destination instead — so the file ends up where you actually keep things.

Status Proposal — not built yet Rollout 3 stages Off by default Yes

What it does

In one sentence.

When you start a download, instead of choosing a folder on your Mac, you choose a server. The file is delivered there.

There are two ways to get it there, and they behave quite differently. Which one you get depends on what you're downloading.

Way 1 — Pass it along

The file downloads to your Mac as normal, gets checked, then goes to the server, and the local copy is removed.

Good: full speed, can be paused and resumed, virus-checked before it reaches your server.
Cost: needs free space on your Mac for a while.

Way 2 — Send it straight through

The file never touches your Mac's disk at all. It passes through and goes straight to the server.

Good: needs almost no space on your Mac.
Cost: slower, cannot be paused, and if the connection drops it starts again from zero.

Internet → Your Mac (briefly) → Your server — Way 1, the normal path

The honest bit "It never touches my Mac" sounds like the better option, but it is the weaker one. It only works for simple downloads, it is slower, you cannot pause it, and it skips the virus check. Way 1 is the recommended default for almost everything.

What we decided

Scope for the first release.

QuestionDecision
What can be sent to a server? Normal link downloads — web links, FTP, video streams, and server-to-server transfers. Torrents are excluded for now.
Where do you choose the server? Two places, added in stages: right-click a finished download first, then a picker when you start one.
Can it run automatically in the background? No. Only when you're there to see it. Watch folders, RSS, the phone remote and the Linux version all stay switched off.
Is the local copy deleted? Not unless you tick a box. Sending is a copy, not a move.
Is it on by default? No. It is off until you turn it on in Settings.

Why leaving torrents out matters Torrents were the messiest part by a wide margin — thousands of small files, and the awkward question of what happens to sharing when the local copy is deleted. Removing them makes the first release dramatically simpler and safer.

How it would be rolled out

Three stages, each useful on its own.

1

Right-click → “Send to server” First

Works on a download you've already finished. No new options when adding downloads, nothing changes about how downloading works. This is the safest possible starting point — it proves the sending part works reliably against real servers before anything else is built on top of it.

2

Choose a server when starting a download Next

The picker in the Add-download window. Downloads land in a temporary holding folder, then go to the server automatically. This is the feature as you originally described it.

3

Send straight through, no local copy Last

The zero-space option. Deliberately last, because it is the trickiest part to get right and the easiest to get subtly wrong.

Why this order Nearly all of the difficult safety work sits in stage 1. Getting it right in the smallest, simplest release means stages 2 and 3 are built on something already proven, rather than everything being tested at once.

What could go wrong

The problems worth knowing about, and what stops each one.

ProblemWhat we do about it
Someone impersonating your server
Your files could be sent to the wrong machine.		 Every server's identity is remembered the first time you connect and checked every time after. If it doesn't match, the transfer stops completely — and there is deliberately no "continue anyway" button.
The phone remote being used to write files to sensitive places on your server
The most serious issue found.		 The remote control feature is never allowed to choose a server destination. Not restricted — not allowed at all.
Finished downloads disappearing from your list
A background tidy-up would have removed them.		 Found during review. That tidy-up gets taught to recognise files that live on a server rather than on your Mac.
Your Mac's disk filling up
If your server is offline, files pile up waiting.		 Space is checked before a download starts, and there's a limit on how much can be held waiting. Past that limit, new downloads wait rather than start.
Your server running out of space mid-transfer Space is checked first where possible. If it happens anyway, the partial file is clearly named and you're offered a one-click cleanup.
A half-finished file looking complete Files are sent under a temporary name and only renamed once they arrive in full. An interrupted transfer always looks obviously unfinished.
Losing your download if the server rejects you
Wrong password, server moved, etc.		 The downloaded file stays safely on your Mac. Retrying re-sends it — it never downloads again from scratch.
Hammering a server that's down
Repeated attempts can get you blocked.		 Retries slow down progressively, and after repeated failures that server is paused until you say otherwise.

The full technical review covers 52 scenarios and 28 specific protections. The eight above are the ones with real consequences.

The off switch

One setting controls the whole feature.

Settings → “Send downloads to a server”

A single switch, off when you install. Until you turn it on, nothing about the app changes — no new buttons, no new options, and nothing is ever sent anywhere.

If you turn it off again later, the rule is simple and absolute: no file on your Mac is ever deleted, and nothing is ever sent to a server. Anything already in progress finishes cleanly or stops cleanly — it is never abandoned halfway. Your saved server choices are kept, so turning it back on picks up where you left off.

Why off by default This feature moves your files to another machine and can delete the local copy. That is not something that should ever switch itself on. Every comparable option in the app already works this way.

What we're deliberately not doing

Just as important as what we are.

Not includedWhy
Torrents Excluded By far the messiest case. Worth revisiting once the rest is proven.
Automatic background sending Excluded Watch folders and RSS run with nobody watching, so they can't answer a security question if one comes up. Too risky for a first release.
Starting from your phone Excluded This was the most serious security issue found in the review.
The Linux version Excluded Doubles the work. Mac only for now.
Dropbox, Google Drive, S3 and similar Excluded Different technology entirely. This is about your own servers.

Two honest limitations

Neither can be fixed by being careful — they're built into how the underlying technology works.

1. Sending to a server cannot be resumed If a transfer to your server is interrupted, it starts again from the beginning. For a very large file over an unreliable connection, this could mean it never finishes.

The download itself resumes normally — this only affects the final hop to the server. Fixing it means changing a low-level part of the app, which isn't in any of the three stages above.

2. We can confirm the size, not the contents After sending, we check the file on the server is the right size. That catches an interrupted transfer, but it can't prove every byte is identical. Proving that would need the ability to run commands on your server, which is out of scope.
